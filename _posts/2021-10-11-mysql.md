# 01 基础架构：一条SQL查询语句是如何执行的

![image.png](/assets/images/mysql1.png)

MySQL可以分为**Server**层和**存储引擎**层两部分。

* Server层包括连接器、查询缓存、分析器、优化器、执行器等，涵盖MySQL的大多数核心服务功能，以及所有的内置函数（如日期、时间、数学和加密函数等），所有跨存储引擎的功能都在这一层实现，比如存储过程、触发器、视图等。不同的存储引擎共用一个Server层。
* 存储引擎层负责数据的存储和提取。其架构模式是插件式的，支持InnoDB、MyISAM、Memory等多个存储引擎。现在最常用的存储引擎是InnoDB，它从MySQL 5.5.5版本开始成为了默认存储引擎。

## 连接器

连接命令中的mysql是客户端工具，在完成经典的TCP握手后，连接器就要开始认证你的身份，如果用户名密码认证通过，连接器会到权限表里面查出你拥有的权限。一个用户成功建立连接后，即使你用管理员账号对这个用户的权限做了修改，也不会影响已经存在连接的权限。

连接完成后，如果你没有后续的动作，这个连接就处于空闲状态，用`show processlist`命令可以看到Command是`Sleep`的一行。客户端如果太长时间没动静，连接器就会自动将它断开。这个时间是由参数`wait_timeout`控制的，默认值是8小时。

建立连接的过程通常是比较复杂的，尽量使用长连接，一个连接可以处理多个请求。但是MySQL在执行过程中临时使用的内存是管理在连接对象里面的，只在断开时才释放，容易占用大量内存。解决方法是：

* 定期断开长连接
* 在大操作之后执行`mysql_reset_connection`来初始化连接资源(版本>=v5.7)

## 查询缓存

查询缓存（key: 查询语句, value: 查询结果），有则返回。(此功能在版本>=8.0上被砍掉了)

一个表有更新，其上所有查询缓存都会被清，所以缓存的失效非常频繁。可以默认让SQL都不用查询缓存：把`query_cache_type`设置成`DEMAND`，想用缓存的时候可以这样写：

> select SQL_CACHE * from T where x=y;

你会在数据库的慢查询日志中看到一个rows_examined的字段，表示这个语句执行过程中扫描了多少行。这个值就是在执行器每次调用引擎获取数据行的时候累加的。

引擎扫描行数跟rows_examined并不是完全相同的。

# 02 日志系统：一条SQL更新语句是如何执行的

`WAL`的全称是`Write-Ahead Logging`，它的关键点就是先写日志，再写磁盘。当有一条记录需要更新的时候，InnoDB引擎就会先把记录写到**redo log**里面（文件名是ib_logfile+数字），并更新内存，这个时候更新就算完成了。同时，InnoDB引擎会在适当的时候，将这个操作记录更新到磁盘里面。redo log是InnoDB引擎特有的日志。

InnoDB的redo log是固定大小的，比如可以配置为一组4个文件，每个文件的大小是1GB，那么总共就可以记录4GB的操作。从头开始写，写到末尾就又回到开头循环写，如下面这个图所示。

![image.png](/assets/images/mysql2.png)

`write pos`是当前记录的位置，一边写一边后移，写到第3号文件末尾后就回到0号文件开头。`checkpoint`是当前要擦除的位置，也是往后推移并且循环的，擦除记录前要把记录更新到数据文件。

有了redo log，InnoDB就可以保证即使数据库发生异常重启，之前提交的记录都不会丢失，这个能力称为**crash-safe**。

Server层也有自己的日志，称为**binlog**。最开始MySQL里并没有InnoDB引擎。MySQL自带的引擎是MyISAM，但是MyISAM没有crash-safe的能力，binlog日志只能用于归档。而InnoDB是另一个公司以插件形式引入MySQL的，既然只依靠binlog是没有crash-safe能力的，所以InnoDB使用另外一套日志系统——也就是redo log来实现crash-safe能力。

* redo log是物理日志，binlog是逻辑日志。
* redo log是循环写的，空间固定会用完；binlog是可以追加写入的。所以只有binlog能用来做归档。
* **有了redo log，系统就crash-safe了**，但也不能没有binlog，它俩用处不一样，很多系统依赖于binlog，比如**MySQL主从复制**，还有一些异构系统，比如搜索服务就监控binlog来更新搜索index。
* **只有binlog没有redo log的话是不能crash-safe的**，binlog没有恢复数据页的能力。说白了binlog不是WAL的，binlog写了，MySQL不知道数据页更没更新。那能不能把binlog做成crash-safe的，能，但其实就是做了个redo-log出来，没必要。

`update T set c=c+1 where ID=2;`的执行过程：

![image.png](/assets/images/mysql3.png)

这个图里的commit和begin/start transaction然后commit的commit语句不是一个东西，这个commit是事务提交过程中的最后一小步，commit语句包含这个小commit步骤。这个例子中没有显式地开启事务，因此这个update语句自己就是一个事务，在执行完成后提交事务时，就会用到这个“commit步骤“，当这个步骤执行完成后，这个事务就提交完成了。

一个事务中有多个语句时，生成的日志都要保存起来，但又不能在还没commit的时候就直接写到redo log文件里，所以有了**redo log buffer**的概念，它就是一块内存，用来先存redo日志的，最后commit了才真正写到redo log文件。

在崩溃恢复场景中，InnoDB如果判断到一个数据页可能在崩溃恢复的时候丢失了更新，就会将它读到内存，然后让redo log更新内存内容。更新完成后，内存页变成脏页。最终数据落盘，刷的就是内存里的脏页。

可以看到redo log的写入拆成了两个步骤：prepare和commit，这就是"**两阶段提交**"，那为啥要两阶段提交呢？

* 如果先写redo log后写binlog：如果在redo log写完，binlog还没写完的时候crash。则主库上redo log能把数据恢复，但备库会因为没有相应的binlog无法恢复数据而导致主备不一致。
* 如果先写binlog后写redo log：如果在binlog写完之后crash，由于redo log还没写，则情况正好相反，备库能恢复出来数据，而主库会因为事务无效而回滚，导致主备不一致。

**redo log和binlog都可以用于表示事务的提交状态，而两阶段提交就是让这两个状态保持逻辑上的一致。**

* 如果在写入prepare后，写binlog之前crash：redo log没commit，事务会回滚，binlog也没写不会传到备库，数据一致。
* 如果在binlog写完，redo log commit之前crash：则按下面规则恢复：
  1. 如果redo log里面的事务是完整的，也就是已经有了commit标识，则直接提交；
  2. 如果redo log里面的事务只有完整的prepare，则判断对应的事务binlog（**redo log和binlog有一个共同的数据字段，叫XID**。崩溃恢复的时候，会按顺序扫描redo log：如果碰到只有parepare、而没有commit的redo log，就拿着XID去binlog找对应的事务。）是否存在并完整，如果是，则提交事务；否则，回滚事务。那MySQL怎么知道binlog是完整的?个事务的binlog是有完整格式的：**statement格式的binlog，最后会有COMMIT；row格式的binlog，最后会有一个XID event。**
  
# 03 事务隔离：为什么你改了我还看不见

ACID，事务隔离就是其中的I（Isolation）。

SQL标准的事务隔离级别包括：读未提交（read uncommitted）、读提交（read committed）、可重复读（repeatable read）和串行化（serializable ）。MySQL默认是RR，Oracle默认是RC。可以查询参数`transaction_isolation`来查看当前隔离级别。

* RU读未提交：一个事务还没提交时，它做的变更就能被别的事务看到。
* RC读提交：一个事务提交之后，它做的变更才会被其他事务看到。
* RR可重复读：一个事务执行过程中看到的数据，总是跟这个事务在启动时看到的数据是一致的。当然在可重复读隔离级别下，未提交变更对其他事务也是不可见的。
* S串行化，顾名思义是对于同一行记录，“写”会加“写锁”，“读”会加“读锁”。当出现读写锁冲突的时候，后访问的事务必须等前一个事务执行完成，才能继续执行。

|隔离级别|脏读|不可重复读|幻读|
|---|---|---|----|
|未提交读|可能|可能|可能|
|已提交读|不可能|可能|可能|
|可重复读|不可能|不可能|可能|
|可串行化|不可能|不可能|不可能|

RR和RC在实现上，数据库里面会创建一个视图，访问的时候以视图的逻辑结果为准。

* RU：直接返回记录上的最新值，没有视图。
* RC：视图是在每个SQL语句开始执行的时候创建的。
* RR：视图是在事务启动时创建的，整个事务存在期间都用这个视图。
* S：直接用加锁的方式来避免并行访问。

**RR是通过read-view实现的**，不同时刻启动的事务会有不同的read-view，假设一个值从1被按顺序改成了2、3、4，在回滚日志里面就会有类似下面的记录：

![image.png](/assets/images/mysql4.png)

对于read-view A，要得到1，就必须将当前值依次执行图中所有的回滚操作得到，**一条记录可以存在多个版本，这个概念就是多版本并发控制（MVCC）**。不要把这个read-view和creat view的那个视图搞混了，这是两个东西，read-view没有物理结构，只是事务执行期间用来定义“能看到什么数据”的。

当系统里没有比这个回滚日志更早的read-view的时候会把这个回滚日志删除，因为不需要了，没有事务再需要用它了。这意味着事务越长，积累的undo log就越大，长事务还占用锁资源，还会导致主从延迟，**综上建议尽量不要使用长事务**。如果`autocommit=0`，则任何语句就自动启动了一个事务，在你主动执行commit或rollback之前都不会提交，建议把`autocommit`设置成1，这样可以用begin或start transaction来显式启动事务，还可以用`commit work and chain`来提交事务并自动启动下一个事务，这样也省去了再次执行begin语句的开销。

可以用下面SQL来查询长事务。

> select * from information_schema.innodb_trx where TIME_TO_SEC(timediff(now(),trx_started))>60

如何避免长事务对业务的影响？

客户端：
* 开启autocommit；
* 只读的事务别用begin/commit；
* 根据业务设置`MAX_EXECUTION_TIME`来控制每个语句最长执行时间；

数据库端：
* 监控information_schema.Innodb_trx表，设置长事务阈值，超过就报警/或者kill；
* 用pt-kill；
* 测试阶段输出general_log，分析日志提前预防。
* 把`innodb_undo_tablespaces`设置成>=2的值，可以undo log会存放在单独的文件而不是ibdata中，方便清理。


# 04 深入浅出索引（上）

在InnoDB中，表都是根据主键顺序以索引的形式存放的，这种存储方式的表称为索引组织表。InnoDB使用了B+树索引模型，数据都是存储在B+树中的。

每一个索引在InnoDB里面对应一棵B+树。

```sql
CREATE TABLE T (
  id INT PRIMARY KEY,
  k INT NOT NULL,
  NAME VARCHAR ( 16 ),
  INDEX ( k )
) ENGINE = INNODB;
```

表中R1~R5的(ID,k)值分别为(100,1)、(200,2)、(300,3)、(500,5)和(600,6)，两棵树的示例示意图如下。

![image.png](/assets/images/mysql5.png)

根据叶子节点的内容，索引类型分为**主键索引和非主键索引**。

**主键索引的叶子节点存的是整行数据**。在InnoDB里，主键索引也被称为**聚簇索引**（clustered index）。

**非主键索引的叶子节点内容是主键的值**。在InnoDB里，非主键索引也被称为**二级索引**（secondary index）。

* `select * from T where ID=500`只需查B+树ID。
* `select * from T where k=5`需要先查树k，得到ID后再查树ID，这个过程称为**回表**，要查两次。**所以尽量使用主键**。
* `select id from T where k=5`查的就只有id，不需要回表，这个叫覆盖索引。

在数据页中间（比如ID=400）插入新数据时如果数据页已经满了，则需要申请新数据页，然后移动部分数据过去，这个叫**页分裂**，即影响性能，也影响空间利用率。分裂过程的逆过程叫**合并**。

**自增主键有利于新增数据**，因为相当于append操作，没有页分裂。只有一个唯一索引的表可以不用自增主键。

**主键长度越小，普通索引的叶子节点就越小，普通索引占用的空间也就越小。**

* 重建普通索引可以先drop后add索引；
* 重建主键索引不能先drop后add，因为这两个都会重建整个表，相当于白做了一遍。可以用`alter table T engine=InnoDB`来重建。

# 05 深入浅出索引（下）

* **最左前缀**：联合索引的最左 N 个字段，也可以是字符串索引的最左 M 个字符；
* **联合索引**：根据创建联合索引的顺序，以最左原则进行where检索，比如（age，name）以age=1 或 age= 1 and name=‘张三’可以使用索引，单以name=‘张三’ 不会使用索引，考虑到存储空间的问题，还请根据业务需求，将查找频繁的数据进行靠左创建索引；建联合索引时，索引建的顺序安排的建议如下：
  * 区分度最高的建在索引最左侧
  * 使用最为频繁的列建在最左侧
  * 长度小的字段放在最左侧
* **索引下推**（index condition pushdown）：like 'hello%’and age >10 检索，MySQL5.6版本之前，会对匹配的数据进行回表查询。5.6版本后，会先过滤掉age<=10的数据，再进行回表查询，减少回表率，提升检索速度；
